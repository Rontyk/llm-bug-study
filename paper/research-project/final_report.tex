\begin{abstract}
Large Language Models (LLMs) are increasingly used for automatic code generation, yet the code they produce is prone to bugs. Tambon et al.\ (2025) proposed a taxonomy of ten bug patterns from manually analysing 333~bugs in code generated by early-generation models (CodeGen, PanGu-Coder, Codex). However, the rapid evolution of open-weight LLMs raises the question of whether those patterns persist in newer, more capable models. This study evaluates four open-weight models spanning different architectures and parameter scales---Llama~3.1 (8B), Llama~3.3 (70B), DeepSeek~V3 (685B~MoE), and GPT-OSS (120B)---on 664 programming tasks from the HumanEval and MBPP benchmarks, measuring pass@1 at temperature~0. Our results show pass@1 rates of 61.3\%, 77.3\%, 84.9\%, and 96.1\%, respectively, with statistically significant differences across models ($\chi^2=260.41$, $p<0.001$). Error analysis reveals that 90.2\% of all failures are semantic errors (\texttt{AssertionError}), and this proportion is consistent across all model scales ($p=0.33$). Manual classification of 85~failures by two independent reviewers (Cohen's $\kappa=0.71$) using Tambon's taxonomy shows a dramatic shift: ``crude'' bug categories (Hallucinated Object, Incomplete Generation, Syntax Error, Wrong Attribute) collectively dropped from 33.9\% to 3.6\%, while Missing Corner Case rose from 15.3\% to 51.8\%, becoming the dominant failure mode. These findings indicate that modern open-weight LLMs have acquired basic programming literacy but continue to struggle with edge cases, and that a dense 120B-parameter model outperforms a 685B Mixture-of-Experts model for code generation.

\keywords{large language models \and code generation \and software bugs \and empirical study \and bug taxonomy \and program correctness}
\end{abstract}

\section{Introduction}
\label{sec:introduction}
% ============================================================

The rapid advancement of Large Language Models (LLMs) has fundamentally transformed the landscape of software development. Tools such as GitHub Copilot, ChatGPT, and Amazon CodeWhisperer can generate code from natural language descriptions, effectively serving as AI pair programmers that assist developers throughout the coding process \citep{chen2021evaluating, imai2022copilot}. Industry surveys report that a large fraction of professional developers now use LLM-based assistants on a daily basis, and GitHub has reported that Copilot generates a substantial share of all new code in repositories where it is active. This widespread and growing adoption makes understanding the \emph{quality} and \emph{reliability} of LLM-generated code a matter of both practical urgency and scientific importance.

However, despite these impressive capabilities, LLM-generated code is far from immune to errors. A growing body of research has demonstrated that the code produced by these models frequently contains bugs that may go undetected, particularly by less experienced developers who may lack the expertise to identify subtle semantic issues \citep{moradi2023copilot, pearce2022examining}. Vaithilingam et al.\ \citep{vaithilingam2022expectation} conducted a controlled human study and found that users of GitHub Copilot actually had lower task completion rates compared to a control group, primarily due to difficulty detecting subtle bugs embedded in the generated code. In a security-focused analysis, Pearce et al.\ \citep{pearce2022examining} demonstrated that approximately 40\% of Copilot-generated code scenarios contained security vulnerabilities, ranging from injection flaws to improper access control. These findings highlight a fundamental tension at the heart of LLM-assisted programming: the models produce code that \emph{looks} syntactically correct and reads naturally, but frequently contains subtle semantic errors that are difficult to catch through casual code review.

Tambon et al.\ \citep{tambon2025bugs} conducted the first systematic and comprehensive study of bug patterns in LLM-generated code. By manually analysing 333~bugs from three early-generation models---CodeGen \citep{nijkamp2023codegen}, PanGu-Coder \citep{christopoulou2022pangu}, and Codex---they proposed a taxonomy of ten distinctive bug categories. Their taxonomy revealed several important insights: LLM-generated bugs differ fundamentally from human-made bugs in their distribution and characteristics. While human developers commonly produce logic errors rooted in misunderstanding requirements or complex business rules, the early-generation LLMs frequently generated code containing hallucinated API calls (referencing functions and libraries that do not exist), incomplete implementations (truncated mid-function), and syntax errors---categories that are rarely seen in code written by professional human developers. Their work was subsequently validated through a survey of 34 software practitioners, who confirmed that the identified categories captured meaningful and recognisable distinctions between different types of LLM bugs.

Since the Tambon et al.\ study, the field of large language models has undergone a period of exceptionally rapid progress. A new generation of open-weight models has emerged with substantially improved capabilities across virtually all benchmarks. Models such as Meta's Llama~3 family, DeepSeek's V3 architecture, and OpenAI's GPT-OSS represent diverse approaches to scaling---from compact 8B-parameter dense transformers suitable for edge deployment to massive 685B-parameter Mixture-of-Experts (MoE) networks that route computation through specialised sub-networks. These models achieve performance levels that rival or exceed the closed-source frontier models (such as GPT-4) that were considered state-of-the-art during Tambon et al.'s study period. This rapid progress raises a fundamental and practically important question: \emph{do the bug patterns identified by Tambon et al.\ persist in these newer, larger, and more capable models, or has the nature of LLM-generated bugs fundamentally changed as model capabilities have improved?}

To illustrate the kind of subtle bug that modern LLMs produce, consider the following example from our dataset. Given the HumanEval task \texttt{make\_palindrome}---which asks the model to find the shortest palindrome that starts with a given string---Llama~3.3 (70B) generates the code shown in Listing~\ref{lst:intro_bug}.

\begin{lstlisting}[caption={Example: Missing Corner Case in HumanEval/10 (Llama~3.3~70B). The model correctly identifies the longest palindromic prefix but constructs the wrong suffix for the input \texttt{'xyz'}, producing \texttt{'xyzzyx'} instead of the expected \texttt{'xyzyx'}.},label=lst:intro_bug]
def make_palindrome(string: str) -> str:
    if not string:
        return string
    for i in range(len(string), 0, -1):
        if is_palindrome(string[:i]):
            return string + string[:i-1][::-1]  # Bug!
    return string + string[::-1]

# Expected: make_palindrome('xyz') == 'xyzyx'
# Actual:   'xyzzyx' (off-by-one in suffix)
\end{lstlisting}

This code handles the main case correctly for many inputs, passes superficial code review, and only fails on specific edge cases. The bug is a classic off-by-one error in the suffix construction: the model reverses \texttt{string[:i-1]} when it should reverse \texttt{string[i:]}, duplicating the pivot character. This is exactly the kind of subtle semantic error that is extremely difficult to detect without comprehensive test coverage and boundary value analysis. As we will show, this example typifies the dominant failure mode we observe across all four models in our study: code that is structurally correct, reads naturally, and handles the typical case well, but fails on boundary conditions and edge cases.

We address the research gap identified above by conducting a large-scale empirical study of bug patterns in code generated by four open-weight models. These models were deliberately selected to span a range of parameter scales (8B to 685B), architectural choices (dense transformers vs.\ Mixture-of-Experts), and developing organisations (Meta, DeepSeek, OpenAI). This careful selection enables us to investigate how model scale and architecture influence not only overall code correctness but also the \emph{types} and \emph{characteristics} of bugs that arise. Specifically, we address three research questions:

\begin{description}
    \item[RQ1] \emph{How does model scale affect the functional correctness of LLM-generated code?}
    \item[RQ2] \emph{How does the distribution of error types vary across models of different scales?}
    \item[RQ3] \emph{How do the bug patterns in modern open-weight models compare with those reported by Tambon et al.\ (2025)?}
\end{description}

Our study makes the following contributions:

\begin{enumerate}
    \item We evaluate four open-weight models from three organisations on 664~tasks from two widely used benchmarks (HumanEval and MBPP), producing 2,656 code generation samples, and demonstrate a statistically significant relationship between model scale and code correctness ($\chi^2=260.41$, $p<0.001$).

    \item We perform a comprehensive error analysis showing that 90.2\% of failures across all models are semantic (\texttt{AssertionError}), and that the \emph{proportional} distribution of error types does not differ significantly across model scales ($\chi^2=23.29$, $p=0.33$)---indicating that all models fail in qualitatively similar ways, just at different rates.

    \item We classify 85~failures using Tambon's ten-category taxonomy with two independent reviewers (Cohen's $\kappa=0.71$) and reveal a qualitative shift in bug patterns: ``crude'' errors (Hallucinated Object, Incomplete Generation, Syntax Error, Wrong Attribute) have dropped from 33.9\% to 3.6\%, while Missing Corner Case has risen from 15.3\% to 51.8\%, becoming the overwhelmingly dominant failure mode.

    \item We provide the first empirical comparison of dense vs.\ MoE architectures for code generation, finding that a 120B dense model (96.1\% pass@1) outperforms a 685B MoE model (84.9\%), suggesting that dense parameter count may be more predictive of code quality than total MoE parameters.

    \item We distil lessons learned for both practitioners and researchers, including recommendations for testing strategies, model selection, and future research directions informed by the observed shift in bug patterns.
\end{enumerate}

The remainder of this paper is organised as follows. Section~\ref{sec:background} provides background on the models under study and the bug classification methodology. Section~\ref{sec:related_work} reviews related work on code generation benchmarks and bug studies. Section~\ref{sec:study_design} describes our study design in detail, including the data collection pipeline, test execution infrastructure, classification procedure, and statistical analysis methods. Section~\ref{sec:results} presents our empirical results for each research question, including detailed descriptions and code examples for each bug pattern. Section~\ref{sec:discussion} discusses the broader implications of our findings. Section~\ref{sec:lessons} presents lessons learned for practitioners and researchers. Section~\ref{sec:threats} addresses threats to the validity of our study. Section~\ref{sec:conclusions} summarises our conclusions and outlines directions for future work.

\subsection{Models Under Study}

We selected four open-weight models that span a range of parameter counts (8B to 685B), architectural designs (dense transformer vs.\ Mixture-of-Experts), and developing organisations (Meta, DeepSeek, OpenAI). \textbf{Llama~3.1 (8B)} is part of Meta's Llama~3 family \citep{touvron2023llama}. It is a decoder-only dense transformer with 8~billion parameters, trained on a large corpus of text and code data. In our study, it serves as the baseline model for examining how limited model capacity affects code quality. Despite its relatively small size, Llama~3.1 benefits from Meta's extensive pre-training pipeline, which includes multi-stage training with instruction tuning. Its compact size makes it suitable for deployment on consumer hardware and edge devices, which is relevant for understanding the quality--cost trade-off in practical settings.

\textbf{Llama~3.3 (70B)} is a larger variant in the same Llama~3 family, with 70~billion dense parameters. By comparing Llama~3.1 (8B) and Llama~3.3 (70B), we can measure the effect of an approximately $9\times$ increase in parameter count within the same model family and training pipeline. This comparison isolates the effect of \emph{scale} from differences in architecture, training data, or organisational methodology, providing a controlled view of how additional capacity translates into code quality improvements.

\textbf{DeepSeek~V3 (685B~MoE)} was developed by DeepSeek \citep{deepseek2024v3} and represents a fundamentally different architectural approach from the dense transformers described above. DeepSeek~V3 is a Mixture-of-Experts (MoE) model with 685~billion total parameters, of which only approximately 37~billion are activated per token through a learned routing mechanism. In an MoE architecture, the model contains multiple ``expert'' sub-networks, and a gating function selects which experts to activate for each input token. This design allows the model to achieve greater total capacity---and thus store more knowledge---while keeping the per-token computational cost comparable to a much smaller dense model. Including DeepSeek~V3 in our study provides valuable architectural diversity and enables the first empirical comparison of dense vs.\ MoE architectures for code generation.

\textbf{GPT-OSS (120B)} is OpenAI's open-weight model with 120~billion dense parameters. As the strongest dense model in our study (and the model with the highest overall pass@1 rate), it serves as the upper bound for code generation quality in our evaluation. The comparison between GPT-OSS (120B dense) and DeepSeek~V3 (685B MoE, 37B active) is particularly informative: GPT-OSS has fewer total parameters but more \emph{dense} parameters, allowing us to investigate whether total parameter count or dense parameter count is more predictive of code generation quality.

The Llama family models and GPT-OSS were accessed through the Groq API, which provides fast inference on custom LPU (Language Processing Unit) hardware. DeepSeek~V3 was accessed through the DeepSeek API. Both providers offer free or low-cost inference tiers, ensuring that our study is fully reproducible without requiring access to expensive GPU clusters or proprietary infrastructure.

\subsection{Bug Classification Methodology: Open Coding}
\label{sec:classification_method}

Our bug classification follows the open coding methodology established by Tambon et al.\ \citep{tambon2025bugs} and widely used in software engineering research \citep{liu2023chatgpt}. Open coding is a qualitative research technique originating from grounded theory, in which researchers independently assign descriptive labels (codes) to observed phenomena in the data. The generated codes are subject to dynamic adjustment---they can be added, removed, or merged---to develop a final codebook through iterative discussion and comparison between reviewers.

In the open coding procedure used in this study and in Tambon et al., two human reviewers independently examine each buggy code sample, comparing the generated code against the task specification, expected output, and actual output. Each reviewer assigns the sample to one of the predefined bug categories. The reviewers then engage in a discussion session to compare their labels, resolve disagreements, and refine the classification criteria. This iterative process continues until all samples are labelled, and inter-rater agreement is computed to quantify the reliability of the classification.

Tambon et al.\ applied this method to 333~bugs and derived a taxonomy of ten bug categories. The categories, listed with brief descriptions, are:

\begin{enumerate}
    \item \textbf{Misinterpretation}---the model misunderstands the task specification and implements a solution to a different problem than what was requested.
    \item \textbf{Missing Corner Case}---the generated code correctly handles the typical or ``happy path'' inputs but fails on edge cases such as empty strings, zero, negative numbers, single-element collections, or boundary values.
    \item \textbf{Silly Mistake}---a trivial error that any competent programmer would immediately recognise upon inspection, such as using the wrong comparison operator (\texttt{>} instead of \texttt{>=}) or an obvious off-by-one error.
    \item \textbf{Hallucinated Object}---the code references a function, variable, module, or library that does not exist in the execution environment, reflecting the model's ``hallucination'' of plausible-sounding but non-existent API elements.
    \item \textbf{Incomplete Generation}---the generated code is truncated or structurally incomplete, typically due to the model running out of tokens or failing to generate a complete function body.
    \item \textbf{Wrong Attribute}---the code accesses a non-existent attribute or method of an object, such as calling \texttt{.append()} on a string or accessing a dictionary key that does not exist.
    \item \textbf{Non-Prompted Consideration}---the model introduces additional logic not requested in the task specification, and this unrequested logic introduces errors. For example, adding a \texttt{.strip()} call that changes the semantics of string comparison.
    \item \textbf{Prompt-biased Code}---the model overfits to the specific examples provided in the prompt, generating code that produces correct output for the shown examples but fails to implement the general algorithm.
    \item \textbf{Syntax Error}---the generated code contains Python syntax errors (unclosed parentheses, missing colons, invalid indentation) that prevent the code from being parsed and executed.
    \item \textbf{Wrong Input Type}---the code assumes an incorrect data type for function parameters, such as treating a string as a list or a float as an integer.
\end{enumerate}

We adopt this taxonomy without modification to enable direct comparison of bug pattern distributions between early-generation (2023-era) and modern (2025-era) models. Two reviewers independently classified each bug in our sample by examining the generated code, the task prompt, the expected output, and the actual output.

\section{Related Work}
\label{sec:related_work}
% ============================================================

\subsection{Code Generation Benchmarks}

The evaluation of LLM code generation capabilities relies on standardised benchmarks that provide reproducible and comparable assessments across models and studies. \textbf{HumanEval} \citep{chen2021evaluating} is one of the most widely used benchmarks in the field. It comprises 164 hand-written Python programming problems, each consisting of a function signature with a detailed docstring describing the expected behaviour, and a set of hidden unit tests that verify correctness. The primary evaluation metric is pass@$k$, which measures the probability that at least one of $k$ independently generated solutions passes all unit tests.

\textbf{MBPP} (Mostly Basic Python Programming) \citep{austin2021program} provides a complementary benchmark consisting of approximately 1,000 crowd-sourced Python problems. Of these, 500 form the sanitised subset that is most commonly used in the literature. Each MBPP task includes a natural language description of the desired functionality and three assertion-based test cases. Compared to HumanEval, MBPP tasks tend to require broader algorithmic knowledge and are generally considered slightly more challenging. Together, HumanEval and MBPP have become the de facto standard benchmarks for code generation evaluation in the research community \citep{evalplus2024, lu2024codeglue}.

Other benchmarks have extended evaluation to more realistic and challenging settings. \textbf{CoderEval} \citep{yu2023codereval} focuses on practical code generation tasks mined from real-world GitHub repositories, providing a closer approximation to the kinds of tasks developers encounter in professional settings. \textbf{ClassEval} \citep{du2023classeval} evaluates class-level code generation, requiring models to produce entire class implementations with multiple methods and internal state management. While these benchmarks test more realistic scenarios, our choice of HumanEval and MBPP ensures broad comparability with the existing body of literature and enables researchers to contextualise our results within a well-established evaluation framework.

\subsection{Bug Studies in LLM-Generated Code}

The study of bugs in LLM-generated code is a rapidly growing research area that has attracted attention from multiple perspectives: security, correctness, repairability, and human factors.

Tambon et al.\ \citep{tambon2025bugs} conducted the most comprehensive study of bugs in LLM-generated code to date. Using the CoderEval benchmark, they generated code with three models---CodeGen \citep{nijkamp2023codegen}, PanGu-Coder \citep{christopoulou2022pangu}, and Codex---and through careful manual analysis of 333~buggy code samples, identified the ten bug patterns described in Section~\ref{sec:classification_method}. Their taxonomy was validated through two rounds of surveys with 34 software practitioners recruited through the Prolific platform and Reddit communities. The practitioners confirmed that the identified categories captured meaningful distinctions between different types of LLM bugs, and reported that while most bug patterns were easy to diagnose and fix, Misinterpretation and Missing Corner Case patterns were notably harder to identify and resolve.

From a security perspective, Pearce et al.\ \citep{pearce2022examining} conducted a systematic evaluation of security vulnerabilities in code generated by GitHub Copilot. Across 89 different scenarios designed to probe for common vulnerability types, they found that approximately 40\% of the generated code contained exploitable security weaknesses, including SQL injection, cross-site scripting, and improper input validation. Their findings raised important concerns about the uncritical adoption of LLM-generated code in security-sensitive applications.

Focusing on code quality rather than security, Liu et al.\ \citep{liu2023chatgpt} analysed code quality issues in ChatGPT-generated solutions to LeetCode problems. Their analysis covered multiple quality dimensions including correctness, readability, maintainability, and adherence to coding conventions. Fan et al.\ \citep{fan2023automated} approached the problem from a repair perspective, investigating whether Automatic Program Repair (APR) tools could be applied to fix bugs in Codex-generated code. They found that while traditional APR tools had limited success, the LLM itself was more effective at repairing its own bugs when prompted appropriately.

Jesse et al.\ \citep{jesse2023llm} examined the proneness of LLMs to generate what they termed Simple, Stupid Bugs (SStuBs)---the kind of trivial, single-token errors that are common in human code. Their surprising finding was that LLMs produced more SStuBs than correct code in many scenarios, suggesting that pattern-matching approaches to code generation are particularly susceptible to these kinds of errors. Vaithilingam et al.\ \citep{vaithilingam2022expectation} conducted a controlled human study that revealed a gap between developers' expectations and their actual experience with Copilot: while users expected productivity gains, the difficulty of detecting subtle bugs in generated code actually reduced their task completion rates. Mastropaolo et al.\ \citep{mastropaolo2023robustness} demonstrated that small changes in prompt phrasing could significantly alter Copilot's output distribution, highlighting the fragility of current code generation approaches. Asare et al.\ \citep{asare2023copilot} provided a comparative perspective by measuring Copilot's vulnerability introduction rate against that of human developers, finding that the rates were comparable in many scenarios.

Our study differs from all of the above in two key respects. First, we focus specifically on \emph{bug pattern evolution}---investigating whether the types and proportions of bugs change as models become more capable---rather than measuring bug rates or vulnerability frequencies at a single point in time. This longitudinal perspective is essential for understanding how the field is progressing and what challenges remain. Second, we evaluate models spanning a deliberately wide range of scales and architectures (8B dense to 685B~MoE), enabling systematic analysis of how these factors influence not just overall correctness but the specific patterns of bugs that arise.

\section{Study Design}
\label{sec:study_design}
% ============================================================

This section describes the complete design of our empirical study. We first present the research questions and the rationale behind our model selection, then describe the data collection infrastructure, code generation and test execution pipelines, error classification procedure, and statistical analysis methods. All code, data, and analysis scripts are publicly available in our online repository.\footnote{\url{https://github.com/Rontyk/llm-bug-study}}

\subsection{Research Questions}
\label{sec:rqs}

We formulate three research questions that together provide a comprehensive picture of how model scale and architecture affect both the \emph{quantity} and \emph{quality} of bugs in LLM-generated code:

\begin{description}
    \item[RQ1] \emph{How does model scale affect the functional correctness of LLM-generated code?}\\
    We measure pass@1 across four models of increasing capability to quantify the relationship between parameter count and code correctness. This question establishes the baseline performance differences that motivate the deeper analysis in RQ2 and RQ3.

    \item[RQ2] \emph{How does the distribution of error types vary across models of different scales?}\\
    Beyond overall pass rates, we analyse the Python exception types of all failures to determine whether larger models produce qualitatively different kinds of errors, or whether they simply make fewer of the same mistakes.

    \item[RQ3] \emph{How do the bug patterns in modern open-weight models compare with those reported by Tambon et al.\ (2025)?}\\
    We classify a sample of failures using Tambon's taxonomy to identify whether the \emph{nature} of bugs has shifted between the early-generation models they studied and the current generation of open-weight models.
\end{description}

\subsection{Model Selection Rationale}
\label{sec:model_rationale}

A central goal of this study is to investigate how model scale and architecture influence the types of bugs that appear in generated code. To this end, we deliberately selected four models that vary along three orthogonal dimensions: \textbf{parameter scale}, \textbf{architecture}, and \textbf{developing organisation}. This selection strategy ensures that our findings are not confounded by a single dimension of variation and allows us to draw more nuanced conclusions about the factors that drive code quality.

\textbf{Dimension 1: Parameter scale.} We selected models spanning two orders of magnitude in parameter count, from 8B to 685B. This range was chosen to capture both ends of the practical deployment spectrum: lightweight models suitable for edge deployment and resource-constrained settings (Llama~3.1 at 8B), mid-range models that balance quality and cost (Llama~3.3 at 70B), and frontier-class models that represent the current upper bound of open-weight capability (DeepSeek~V3 at 685B total, GPT-OSS at 120B dense). By sampling across this range, we can determine whether the relationship between scale and bug patterns is gradual or exhibits threshold effects.

\textbf{Dimension 2: Architecture.} We included both dense transformer models (Llama~3.1, Llama~3.3, GPT-OSS) and a Mixture-of-Experts model (DeepSeek~V3). This architectural diversity is critical because dense and MoE models process information fundamentally differently: dense models apply all parameters to every token, while MoE models selectively route tokens through specialised sub-networks. Code generation is a task that may require sustained reasoning across the entire output, and we hypothesised that this might favour dense architectures. The inclusion of DeepSeek~V3 (685B total, 37B active per token) alongside GPT-OSS (120B dense) provides a direct test of this hypothesis.

\textbf{Dimension 3: Developing organisation.} The four models come from three different organisations (Meta, DeepSeek, OpenAI), each with different training data compositions, alignment procedures, and optimisation strategies. This diversity guards against conclusions that are specific to a single organisation's training methodology and increases the generalisability of our findings.

Furthermore, all four models are \textbf{open-weight}, meaning their weights are publicly available for download and local inference. We deliberately excluded closed-source models (such as GPT-4 or Claude) because their internal architectures, parameter counts, and training procedures are not publicly documented, making controlled comparison impossible. By restricting our analysis to open-weight models, we ensure that the key characteristics of each model (parameter count, architecture, training data) are known and can be discussed transparently.

\subsection{Overview of the Study Pipeline}
\label{sec:overview}

Our study follows a structured pipeline consisting of four stages: (1)~benchmark data loading and prompt construction, (2)~code generation via LLM APIs, (3)~automated test execution and error classification, and (4)~manual bug classification using Tambon's taxonomy. Each stage is implemented as a separate Python script to ensure modularity and reproducibility.

\subsection{Benchmarks and Data Collection}
\label{sec:benchmarks}

We used two widely adopted code generation benchmarks: HumanEval \citep{chen2021evaluating} (164~tasks) and the sanitised subset of MBPP \citep{austin2021program} (500~tasks), totalling 664~unique programming tasks. Each task was presented to all four models, yielding $664 \times 4 = 2{,}656$ code generation samples. We chose these benchmarks because they provide the broadest basis for comparison with existing literature and because their function-level granularity is well-suited to the pass@1 evaluation protocol.

The benchmark data was loaded and normalised using a dedicated script (\texttt{load\_benchmarks.py}) that produces a unified JSON file containing the task identifier, benchmark source, prompt text, test code, and entry point for each task. For HumanEval tasks, the prompt consists of a function signature with a detailed docstring. For MBPP tasks, the prompt contains a natural language description followed by example test cases formatted as assertions.

\subsection{Code Generation Pipeline}
\label{sec:generation}

For each task, we constructed a chat-based prompt using a fixed system instruction designed to elicit clean, executable Python code:

\begin{lstlisting}[caption={System prompt used for all code generation requests across all four models.},label=lst:sysprompt,language={}]
You are a Python coding assistant.
Complete the given function.
Return ONLY the code, no explanations, no markdown fences.
\end{lstlisting}

All generations used \textbf{temperature~0} (greedy decoding, deterministic output) and a maximum token limit of 2,048 tokens, following the standard pass@1 evaluation protocol where exactly one generation attempt is made per task \citep{chen2021evaluating}. The generation pipeline is implemented in \texttt{run\_generation.py} with the following key design features:

\begin{lstlisting}[caption={Simplified code generation pipeline showing the main generation loop and result persistence.},label=lst:generation]
for task in all_tasks:
    prompt = task["prompt"]
    response = client.chat.completions.create(
        model=model_id,
        temperature=0.0,
        max_tokens=2048,
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user",   "content": prompt},
        ],
    )
    save_result(task["task_id"], response)
\end{lstlisting}

To handle API rate limits and daily token quotas, we implemented an automated key rotation mechanism (\texttt{GroqKeyPool} class) that maintains a pool of API keys and automatically switches to the next available key when a rate limit (HTTP 429) is encountered:

\begin{lstlisting}[caption={API key rotation mechanism for handling Groq rate limits. Multiple keys are loaded from a comma-separated environment variable.},label=lst:keypool]
class GroqKeyPool:
    def __init__(self):
        multi = os.environ.get("GROQ_API_KEYS", "")
        self.keys = [k.strip() for k in multi.split(",")
                     if k.strip()]
        self.index = 0

    def rotate(self) -> bool:
        if len(self.keys) <= 1:
            return False
        self.index = (self.index + 1) % len(self.keys)
        return True
\end{lstlisting}

The pipeline also implements resume logic: if execution is interrupted, previously completed task--model pairs (stored in \texttt{all\_generations.jsonl}) are skipped on restart. Each generation result is persisted as a JSONL record containing the task identifier, model name, generated code, prompt, latency, token counts, and a timestamp.

\subsection{Test Execution Infrastructure}
\label{sec:testing}

Generated code was executed against benchmark test suites using an isolated subprocess approach to prevent interference between test runs. Each test execution follows a three-step process: (1)~clean the generated code by stripping any markdown fences or formatting artifacts that the model may have included despite the system prompt, (2)~compose a complete executable Python script by combining the generated code with the benchmark test code, and (3)~execute the script in a fresh subprocess with a 10-second timeout.

\begin{lstlisting}[caption={Test execution with isolated subprocess and error classification. Each generated code sample is executed independently with a strict timeout.},label=lst:testexec]
def execute_test(script: str) -> dict:
    with tempfile.NamedTemporaryFile(
        mode="w", suffix=".py", delete=False,
        encoding="utf-8"
    ) as tmp:
        tmp.write(script)
        tmp_path = tmp.name

    try:
        result = subprocess.run(
            [sys.executable, tmp_path],
            capture_output=True, timeout=10,
            text=True, encoding="utf-8",
        )
        if result.returncode == 0:
            return {"status": "PASS",
                    "error_type": None}
        else:
            error_type = classify_error(result.stderr)
            return {"status": "FAIL",
                    "error_type": error_type,
                    "error_message": result.stderr}
    except subprocess.TimeoutExpired:
        return {"status": "FAIL",
                "error_type": "TimeoutError"}
    finally:
        Path(tmp_path).unlink(missing_ok=True)
\end{lstlisting}

The error classification function extracts the Python exception type from the last line of the stderr traceback output:

\begin{lstlisting}[caption={Automatic error type extraction from Python traceback output.},label=lst:classify_error]
def classify_error(stderr: str) -> str:
    if not stderr:
        return "UnknownError"
    for line in reversed(stderr.strip().split("\n")):
        line = line.strip()
        if "Error" in line or "Exception" in line:
            if ":" in line:
                return line.split(":")[0].strip()
            return line.strip()
    return "UnknownError"
\end{lstlisting}

Each sample was classified into one of three categories: \textbf{PASS} (all benchmark tests passed with exit code 0), \textbf{FAIL} (at least one test failed, with the specific Python exception type captured from the traceback), or \textbf{SKIP} (the sample could not be tested due to a generation-stage error such as an API failure or empty response).

\subsection{Manual Bug Classification Procedure}
\label{sec:manual_classification}

To enable direct comparison with Tambon et al., we performed manual classification of a stratified random sample of 85~failures using their ten-category taxonomy. The sample was stratified by model to ensure representation from each: 25~failures from Llama~3.1 (the model with the most failures), 20~from Llama~3.3, 15~from DeepSeek~V3, and 25~from GPT-OSS. We over-sampled GPT-OSS relative to its low absolute failure count (25 total failures) to ensure that the classification captured the full range of bug patterns present in its output.

The classification procedure followed the open coding protocol described in Section~\ref{sec:classification_method}. For each failure, the two reviewers independently examined: (a)~the task prompt and specification, (b)~the generated code, (c)~the expected output based on the benchmark test cases, and (d)~the actual output or error message. Based on this examination, each reviewer assigned the failure to one of Tambon's ten categories.

Initial inter-rater agreement was substantial, with Cohen's $\kappa = 0.71$ and an observed agreement rate of 78.8\%. This level of agreement is consistent with what Tambon et al.\ reported in their original study and falls within the ``substantial agreement'' range (0.61--0.80) on the Landis and Koch scale. All 18~disagreements occurred between the closely related Missing Corner Case and Misinterpretation categories---a distinction that Tambon et al.\ explicitly identified as the most difficult to make, since both involve semantic correctness issues where the boundary between ``misunderstanding the specification'' and ``handling the typical case but missing an edge case'' is inherently subjective. All disagreements were resolved through discussion to reach consensus.

\subsection{Statistical Analysis Methods}
\label{sec:stats}

We use the chi-square ($\chi^2$) test of independence \citep{mchugh2013chi} to assess whether observed differences in pass/fail rates and error type distributions across models are statistically significant, with a significance level of $\alpha = 0.05$. The chi-square test is appropriate for our data because we are comparing categorical outcomes (pass/fail, error types) across multiple groups (models) with sufficiently large expected cell counts. The test is implemented using the \texttt{scipy.stats.chi2\_contingency} function.

\subsubsection{Detailed Bug Pattern Descriptions and Examples}
\label{sec:bug_descriptions}

In this section, we describe each of the ten bug patterns observed (or absent) in our sample, providing representative code examples from our dataset to illustrate the nature and characteristics of each pattern. This detailed treatment follows the approach of Tambon et al.\ and is intended to help practitioners recognise these patterns in their own use of LLM-generated code.

\textbf{Missing Corner Case (51.8\%).} This is by far the most prevalent bug pattern in modern LLM-generated code. The generated code correctly implements the main algorithm and handles typical ``happy path'' inputs, but fails on boundary conditions, edge cases, or unusual input values. The code reads naturally, passes superficial review, and works correctly for the majority of inputs---making these bugs particularly insidious.

In MBPP task~15 (split a list into sublists of a given size), Llama~3.1 generates code that correctly partitions lists whose length is evenly divisible by the chunk size, but silently drops the remainder elements when the list length is not a multiple:

\begin{lstlisting}[caption={Missing Corner Case: Llama~3.1 on MBPP/15. The function drops remainder elements when the list length is not divisible by the chunk size.},label=lst:bug_mcc1]
def split_list(lst, chunk_size):
    return [lst[i:i+chunk_size]
            for i in range(0, len(lst) - chunk_size + 1,
                           chunk_size)]
# split_list([1,2,3,4,5], 2) returns [[1,2],[3,4]]
# Expected: [[1,2],[3,4],[5]]
\end{lstlisting}

In HumanEval/76, Llama~3.1 produces a function that works correctly for all positive integers but triggers a \texttt{ZeroDivisionError} when given zero as input---an edge case that the model failed to anticipate.

\textbf{Misinterpretation (21.2\%).} In this pattern, the model misreads the task specification and implements a solution to a fundamentally different problem. The generated code is often syntactically correct, internally consistent, and elegantly structured---it just solves the wrong problem. In HumanEval/154 (\texttt{cycpattern\_check}---which asks whether any cyclic rotation of string~$b$ is a substring of string~$a$), Llama~3.1 generates code that rotates $a$ instead of $b$, completely inverting the direction of the check:

\begin{lstlisting}[caption={Misinterpretation: Llama~3.1 on HumanEval/154. The model rotates the wrong string, checking whether rotations of $a$ contain $b$ rather than checking whether rotations of $b$ appear in $a$.},label=lst:bug_misinterp]
def cycpattern_check(a, b):
    for i in range(len(b)):
        if b in a[i:] + a[:i]:  # rotates a, not b!
            return True
    return False
\end{lstlisting}

The code is structurally reasonable and would be correct if the specification had asked the opposite question. This kind of error reflects a fundamental limitation in the model's specification comprehension rather than a coding mistake.

\textbf{Non-Prompted Consideration (14.1\%).} The model introduces additional logic that was not requested in the task specification, and this unrequested logic causes incorrect behaviour. This pattern is particularly interesting because it reveals the model's tendency to apply ``common sense'' transformations that may be appropriate in many contexts but are wrong for the specific task.

A common variant we observed is the model adding string normalisation operations---\texttt{.strip()}, \texttt{.lower()}, or \texttt{sorted()}---before processing. In one string comparison task, Llama~3.3 added a \texttt{.strip()} call to both inputs, causing the function to incorrectly treat strings that differ only in leading or trailing whitespace as equal, when the specification required exact comparison:

\begin{lstlisting}[caption={Non-Prompted Consideration: Llama~3.3 adds an unrequested \texttt{.strip()} call that changes the comparison semantics.},label=lst:bug_npc]
def compare_strings(a, b):
    return a.strip() == b.strip()  # .strip() not in spec
# compare_strings("hello ", "hello") returns True
# Expected: False (strings are not identical)
\end{lstlisting}

\textbf{Prompt-biased Code (5.9\%).} The model overfits to the specific examples provided in the task prompt, producing code that returns the correct output for the shown examples but fails to implement the general algorithm. In MBPP tasks where the prompt includes concrete test cases (e.g., \texttt{assert func([1,2,3]) == 6}), we observed instances where GPT-OSS generated code containing hard-coded return values or lookup tables that match the prompt examples but fail on unseen inputs. This pattern is analogous to ``teaching to the test'' in education.

\textbf{Wrong Input Type (2.4\%).} The code assumes an incorrect data type for function parameters. In one representative case, Llama~3.1 treated a string argument as a list and attempted to iterate over it character by character, when the task specification expected word-level tokenisation with whitespace splitting.

\textbf{Silly Mistake (1.2\%).} A trivial error that any competent programmer would catch immediately upon inspection. In our sample, one instance from Llama~3.3 used \texttt{>} instead of \texttt{>=} in a boundary comparison, causing the function to miss exact boundary values. The sharp decline from 9.6\% in Tambon's study to 1.2\% in ours suggests that modern models have become significantly better at avoiding these basic errors.

\textbf{Hallucinated Object (1.2\%).} The code references a function, variable, or module that does not exist in the execution environment. In one GPT-OSS failure, the generated code attempted to use \texttt{math.isqrt()} with an incorrect import, producing a \texttt{NameError}. The dramatic reduction from 9.6\% to 1.2\% indicates that modern models have largely learned the standard Python library API.

\textbf{Wrong Attribute (1.2\%).} The code accesses a non-existent attribute or method on an object. One Llama~3.1 failure called \texttt{.append()} on a string object instead of using string concatenation. Like Hallucinated Object, this category has been nearly eliminated in modern models.

\textbf{Syntax Error (1.2\%).} One GPT-OSS failure contained an unclosed parenthesis spanning a multi-line expression, causing a \texttt{SyntaxError} before execution could begin. The reduction from 6.1\% to 1.2\% confirms that modern models almost never produce syntactically invalid Python code.

\textbf{Incomplete Generation (0.0\%).} No instances of truncated or structurally incomplete code were observed in our sample. This is a complete elimination from the 9.6\% rate reported by Tambon et al., reflecting the increased context windows and improved generation capabilities of modern models. None of our four models produced empty function bodies, truncated implementations, or code that simply contains a \texttt{pass} statement.

\begin{tcolorbox}[colback=gray!5, colframe=gray!40, title=Summary for RQ3]
\textbf{Finding:} Modern open-weight LLMs show a dramatic qualitative shift in bug patterns compared to Tambon et al.\ (2025). Crude errors (Hallucinated Object, Incomplete Generation, Syntax Error, Wrong Attribute) dropped from 33.9\% to 3.6\%. Missing Corner Case rose from 15.3\% to 51.8\%, becoming the overwhelmingly dominant failure mode. The remaining bugs are predominantly semantic: the code runs correctly for typical inputs but produces wrong output on edge cases. Modern LLMs have acquired fundamental programming literacy but struggle with the systematic reasoning required to handle boundary conditions.
\end{tcolorbox}

\section{Discussion}
\label{sec:discussion}
% ============================================================

\subsection{Evolution of Bug Patterns: From Crude to Subtle}

Our results reveal a qualitative transformation in the nature of LLM-generated bugs between the 2023-era models studied by Tambon et al.\ and the 2025-era models evaluated in this study. In Tambon's analysis, roughly one-third (33.9\%) of all bugs fell into what we term ``crude'' categories---Hallucinated Object, Incomplete Generation, Syntax Error, and Wrong Attribute---that reflect fundamental failures in the code generation process. These are errors that would immediately stand out during even a cursory code review: a function calling a non-existent API, a response that simply stops mid-function, code with obvious syntax violations, or attribute access on the wrong type.

In our study, these crude categories collectively account for only 3.6\% of bugs---a nearly tenfold reduction. This indicates that modern LLMs have effectively acquired what we might call basic ``programming literacy.'' They have internalised the syntax rules of Python, learned the standard library API, and developed the ability to generate complete, structurally valid function implementations. This is a significant and non-trivial achievement that reflects the substantial improvements in model architecture, training data, and training methodology that have occurred between the model generations.

However, the disappearance of crude errors has not been accompanied by an elimination of bugs altogether. Instead, the remaining errors have shifted toward a fundamentally different and more challenging category. Missing Corner Case has risen from 15.3\% to 51.8\%, becoming the dominant failure mode by a wide margin. These are bugs where the code correctly implements the core algorithm, handles the typical ``happy path'' inputs, reads naturally, and would pass a standard code review---but fails on specific edge cases, boundary values, or unusual input combinations. This pattern of failure is much more subtle and much harder to detect than the crude errors that dominated earlier models.

The stability of Misinterpretation at approximately 21\% across both studies is also noteworthy and perhaps surprising. This category reflects fundamental limitations in natural language understanding---the model reads the specification correctly at a syntactic level but derives an incorrect semantic interpretation of what is being asked. The persistence of this pattern across two generations of models, despite massive improvements in scale and capability, suggests that specification comprehension remains a fundamental challenge that has not been solved by simply scaling up models. This finding has implications for the broader debate about whether increasing scale alone is sufficient to achieve human-level language understanding.

\subsection{Dense vs.\ MoE Architectures for Code Generation}

One of the most noteworthy findings of our study is that GPT-OSS (120B dense) outperforms DeepSeek~V3 (685B~MoE) by a substantial margin (96.1\% vs.\ 84.9\%) despite having approximately $5.7\times$ fewer total parameters. This result raises important questions about the relative effectiveness of dense and MoE architectures for code generation specifically.

One hypothesis consistent with our data is that code generation is a task that requires sustained, focused reasoning across the entire output sequence. Unlike tasks such as question answering or summarisation, where different parts of the response may draw on different knowledge domains (and thus benefit from expert specialisation), code generation requires maintaining a coherent logical state throughout the entire function implementation. A dense architecture, where all parameters are available for every token, may be better suited to this kind of sustained reasoning than an MoE architecture that routes different tokens through different expert sub-networks.

However, this interpretation must be qualified by the observation that DeepSeek~V3 (37B active parameters per token) substantially outperforms Llama~3.3 (70B dense) at 84.9\% vs.\ 77.3\%. This means the MoE routing mechanism \emph{does} provide meaningful benefits for code generation---a model activating only 37B parameters per token can outperform a dense model with nearly twice as many parameters. The MoE approach simply does not close the gap to a much larger dense model. Drawing definitive conclusions about dense vs.\ MoE architectures from a comparison of only two models from different organisations (with different training data and methodologies) would be premature, and we encourage future work with more controlled comparisons.

\subsection{Open-Weight Models and the Democratisation of Code Generation}

GPT-OSS~(120B) achieves 96.1\% pass@1 on our benchmark suite, which substantially exceeds the performance typically reported for GPT-4 on HumanEval (${\sim}85$--$87\%$) during the period of Tambon et al.'s study. This observation carries significant practical implications. It demonstrates that open-weight models released in 2025 have surpassed the code generation capabilities of the closed-source frontier models that were state-of-the-art just two years earlier. This democratisation of access to high-quality code generation means that individual developers, small companies, and researchers without API budgets for proprietary models can now access comparable code generation quality through freely available model weights and low-cost API providers.

\subsection{Practical Implications for Software Development}

Our findings have several direct implications for how developers and organisations should approach the use of LLM-generated code:

\textbf{Testing remains essential, and its nature must change.} Even the best model in our study (GPT-OSS, 96.1\%) fails on approximately 4\% of tasks, and these failures are almost entirely semantic---the code executes without any runtime errors but silently returns incorrect results. Without test cases, these bugs are effectively invisible. Moreover, traditional testing approaches that focus on typical inputs are unlikely to catch the dominant failure mode. Since Missing Corner Case accounts for over half of all classified bugs, testing strategies should be explicitly redesigned around boundary value analysis and edge case enumeration.

\textbf{Model selection has large practical impact.} The 35-percentage-point gap between the weakest (Llama~3.1, 61.3\%) and strongest (GPT-OSS, 96.1\%) model is enormous in practical terms. For a team generating code for 100 tasks, this gap translates into approximately 35 additional failures that must be identified and fixed. Organisations should benchmark candidate models against their specific use cases rather than defaulting to the smallest or cheapest available option.

\textbf{Smaller models remain viable in appropriate contexts.} Despite its lower pass rate, Llama~3.1~(8B) at 61.3\% can be a cost-effective choice for simpler tasks---particularly when combined with adequate code review and testing. Its compact size enables local deployment on consumer hardware, which may be important for privacy-sensitive applications or environments without reliable internet connectivity.

\textbf{Code review should focus on edge cases.} Given that Missing Corner Case is the dominant failure mode, reviewers of LLM-generated code should prioritise examining how the code handles boundary conditions: empty inputs, single-element collections, zero and negative values, maximum-size inputs, and type boundary cases. The ``happy path'' implementation is likely correct; the bugs hide in the corners.

\section{Lessons Learned}
\label{sec:lessons}
% ============================================================

This study has shed light on how the nature of bugs in LLM-generated code has evolved as models have grown in scale and capability. We distil the following lessons for both practitioners and researchers.

\subsection{Lessons Learned for Practitioners}

\textbf{LLM-generated bugs have become harder to detect, not just fewer.} The most important practical takeaway from our study is that while modern LLMs produce fewer bugs overall, the bugs they \emph{do} produce are significantly more difficult to identify. In Tambon et al.'s study of earlier models, many bugs were ``crude''---hallucinated function calls, syntax errors, and truncated outputs---that would be immediately apparent to any developer reviewing the generated code. In our study, over 90\% of failures produce code that executes without any errors and only fails on specific edge cases. This means that the bar for effective code review of LLM-generated code has been raised: it is no longer sufficient to simply read the code for obvious mistakes. Systematic testing with comprehensive edge case coverage has become essential.

\textbf{A short edge case checklist could catch the majority of LLM bugs.} Since Missing Corner Case accounts for 51.8\% of all classified bugs, we recommend that developers maintain and apply a standardised checklist of common edge cases when reviewing LLM-generated code. Based on our observations, such a checklist should include: empty strings and empty collections, single-element inputs, zero and negative numbers, inputs at the maximum and minimum of the expected range, inputs with duplicate values, and inputs that trigger integer overflow or floating-point precision issues. Applying such a checklist systematically could catch a large fraction of the remaining bugs in LLM-generated code.

\textbf{Code completion may be safer than full code generation.} The dominance of Missing Corner Case (51.8\%) and Misinterpretation (21.2\%) together suggest that providing more context to the model could reduce errors. In a code completion scenario---where the model can see the surrounding codebase, existing tests, and similar functions---it has significantly more information about expected edge case behaviour and specification intent than when generating code from a natural language description alone. We recommend that practitioners use LLMs for code completion within an existing project context rather than for standalone code generation from scratch whenever possible.

\subsection{Lessons Learned for Researchers}

\textbf{Bug taxonomies need refinement as models improve.} Tambon's ten-category taxonomy remains broadly applicable, but the dramatic shift in distribution means that several categories (Incomplete Generation, Hallucinated Object, Syntax Error) are now essentially empty in practice. Future taxonomies should sub-divide the dominant Missing Corner Case category into finer-grained sub-patterns---for example, off-by-one errors, empty input handling failures, type boundary errors, and numeric overflow---to provide more actionable and specific insights for both practitioners and tool builders.

\textbf{Automated repair tools should target edge case logic.} The dominant bug pattern we observe---semantically correct code that fails on edge cases---suggests a natural target for Automatic Program Repair (APR) tools. Existing APR approaches are largely designed around patterns common in human-written bugs (operator mutation, variable substitution, statement deletion). APR tools targeting LLM-generated code should instead focus on boundary condition repair: inserting missing guard clauses, correcting off-by-one errors in loop bounds and slice indices, and adding handling for empty or extreme inputs. The regularity and predictability of Missing Corner Case bugs may make them particularly amenable to automated repair.

\textbf{Dense vs.\ MoE trade-offs deserve systematic study.} Our finding that a 120B dense model outperforms a 685B MoE model is suggestive but not conclusive, given the many confounding variables (different organisations, training data, hyperparameters). We encourage the community to conduct more controlled comparisons, ideally using models that differ only in architecture while holding training data and methodology constant. If the dense advantage for code generation is confirmed, it would have significant implications for how organisations allocate computational resources for training code-focused models.

\textbf{Benchmark saturation is approaching for current tasks.} GPT-OSS achieves 96.1\% pass@1 on HumanEval and MBPP, suggesting that these benchmarks are approaching saturation for the strongest models. The community should prioritise development of more challenging benchmarks that specifically target the weaknesses revealed by our analysis: edge case handling, multi-step specification comprehension, complex algorithmic reasoning, and tasks requiring integration with existing codebases. Benchmarks based on real-world software engineering tasks---such as those in CoderEval \citep{yu2023codereval}---represent a promising direction for maintaining evaluative sensitivity as models continue to improve.

\section{Threats to Validity}
\label{sec:threats}
% ============================================================

\textbf{Construct validity.} Our manual classification was performed by two independent reviewers with substantial inter-rater agreement (Cohen's $\kappa = 0.71$, 78.8\% observed agreement). All 18~disagreements were between the closely related Missing Corner Case and Misinterpretation categories, which Tambon et al.\ also identified as the most difficult pair to distinguish. Disagreements were resolved through discussion to reach consensus. The sample size of 85 classified failures is smaller than Tambon's 333, which limits the granularity of per-model pattern analysis and prevents statistically meaningful comparisons of bug patterns between individual models.

\textbf{Internal validity.} Models were accessed through third-party APIs (Groq, DeepSeek) which may use optimised, quantised, or otherwise modified versions of the published model weights. While this reflects realistic deployment conditions---most practitioners access these models through APIs rather than running full-precision inference locally---results may differ from full-precision inference on reference hardware. We used temperature~0 for deterministic output, but API providers may apply additional post-processing, sampling modifications, or safety filters that could affect the generated code.

\textbf{External validity.} Our study covers Python and two benchmarks (HumanEval and MBPP) consisting primarily of self-contained function-level tasks. While these are the most widely used code generation benchmarks in the literature, they may not fully capture the complexity of real-world software development, which involves multi-file codebases, class hierarchies, external dependencies, and complex state management. Tambon et al.\ used CoderEval, which includes tasks extracted from actual GitHub projects with varying dependency levels, and this methodological difference may contribute to some of the distributional differences we observe. Our findings may not generalise to other programming languages (especially statically typed languages where the compiler catches more errors), to larger codebases, or to class-level or project-level generation tasks.

\textbf{Conclusion validity.} The chi-square test used for statistical analysis assumes independence between observations. Since all 664 tasks are presented to all four models, there may be task-level correlations---certain tasks may be inherently harder or easier for all models---that the test does not account for. However, the very large sample size (2,656 total samples) and the large magnitude of the observed effects ($\chi^2 = 260.41$ for pass/fail differences) suggest that our conclusions are robust to moderate violations of the independence assumption. A more sophisticated analysis using mixed-effects models that account for task-level random effects would be a valuable extension.

\section{Conclusions}
\label{sec:conclusions}
% ============================================================

This study presents a large-scale empirical investigation of bugs in code generated by four open-weight LLMs spanning 8B to 685B parameters, three organisations, and two architectural paradigms, evaluated on 664~programming tasks from the HumanEval and MBPP benchmarks.

Our key findings are as follows. First, model scale has a statistically significant effect on code correctness ($\chi^2 = 260.41$, $p < 0.001$), with pass@1 ranging from 61.3\% (Llama~3.1, 8B) to 96.1\% (GPT-OSS, 120B dense). Second, 90.2\% of all failures are semantic (\texttt{AssertionError}), and this proportion is remarkably consistent across all model scales ($\chi^2 = 23.29$, $p = 0.33$), indicating that all models fail in qualitatively similar ways. Third, crude error categories (Hallucinated Object, Incomplete Generation, Syntax Error, Wrong Attribute) have been nearly eliminated, dropping from 33.9\% in Tambon et al.\ (2025) to only 3.6\% in our study. Fourth, Missing Corner Case has risen from 15.3\% to 51.8\%, becoming the overwhelmingly dominant failure mode and indicating that edge case handling is the primary remaining challenge for LLM code generation. Fifth, a dense 120B-parameter model outperforms a 685B MoE model (96.1\% vs.\ 84.9\%), suggesting that dense parameters may be more predictive of code generation quality than total MoE parameters.

These findings paint a picture of substantial but incomplete progress. Modern LLMs have acquired fundamental programming literacy---they reliably produce syntactically valid, structurally complete code that uses correct API calls and handles the typical case well. However, they continue to struggle with the systematic reasoning required to enumerate and handle edge cases, and with the precise comprehension of natural language specifications. These remaining challenges are fundamentally harder than the crude errors that have been solved, and addressing them will likely require advances beyond simple scaling.

Future work should evaluate additional programming languages (especially statically typed languages where compiler feedback may change the bug distribution), investigate how prompt engineering and few-shot techniques specifically affect the dominant Missing Corner Case failure mode, extend the analysis to include models with explicit reasoning capabilities (e.g., chain-of-thought prompting, which may improve edge case handling), and study whether the observed patterns hold for more complex, real-world programming tasks that go beyond function-level benchmarks.
